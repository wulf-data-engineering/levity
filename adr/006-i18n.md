# ADR‑006 — Internationalization and Localization (i18n)

**Status**: Proposed  
**Date**: 2026‑04‑13

---

## 1. Design & Technology Decision

We commit to building the application with native support for **polyglot environments** spanning both the client and server sides.

Key characteristics:

| Aspect             | Choice                                                                                                                                           |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| Frontend Library   | **ParaglideJS** (`@inlang/paraglide-js`). We use the base library rather than `paraglide-sveltekit` to ensure full SPA/SSG static compatibility.   |
| Frontend Strategy  | Client-side dynamic loading via `navigator.language` resolution in `+layout.svelte`, avoiding URL subpath routing (`/en/`, `/de/`).                |
| Backend Library    | **rust-i18n** for handling localized generic and error messages on the Rust API side.                                                            |
| Backend Strategy   | Contextual locale inference through `Accept-Language` headers, or user record lookups (when the user is fully authenticated).                    |
| Dictionary Source  | Base English dictionaries (`en.json` on client, `en.yml` on backend) acting as ground truth, augmented automatically by the agent workflows.     |

**Why not `paraglide-sveltekit`?**
Using routing-based adapters generally conflicts with a pure SPA output running on an S3 bucket or causes complicated Lambda@Edge mapping rules. Resolving language locally on the client provides speed and resilience while maintaining our "compile to static PWA" ethos.

---

## 3. Tool-Set Support

- `frontend/messages/` folder containing the application's localized JSON dictionaries.
- `backend/locales/` folder containing the API's localized YML strings.
- Automatically enforced agent instructions within `SKILL.md` enforcing any new feature to extract hardcoded text to dictionaries.
- `.agent/workflows/translate-app.md` workflow for explicitly scaffolding a new translated locale targeting the system.

---

### Consequences

- Developers must extract generic or displayed raw text out of component/api files and place it exclusively in translation files.
- Language isn't rigidly enforced by URL (making locale-specific deep linking mildly harder, but ensuring a user's browser setting effortlessly drives the initial experience).
- Agentic assistants will naturally carry out the translation maintenance overhead.

### Alternatives considered

- **Svelte-i18n** – Excellent reactivity, but lacks the robust compilation-time type safety guaranteed by ParaglideJS.
- **Multiple SPA Builds** – Building a complete separate SPA bundle per language. Bloats CI times and adds enormous complexity to Cloudfront distributions routing.
